# 1. 方案目标

当前诉求表面上是 Integration 页面需要支持 plugin 自定义子 Tab 展示状态、消息列表、会话列表、用户列表等信息，但如果只为 Integration 单独做一套 Dynamic Container Renderer，会造成：

* 能力只服务一个业务域，复用性差
* 未来 Agent、Knowledgebase、Project、Workspace、Sandbox、Connector 等模块也会出现类似需求
* 插件体系会逐步分裂成“配置扩展”“页面扩展”“详情扩展”多套机制
* 前端会积累很多场景专属动态页面逻辑，后期难统一

所以建议直接抽象成一个平台级能力：

---

## 平台级能力名称建议

**Universal Declarative View Extension System**

中文可以叫：

**通用声明式扩展视图系统**

它的目标不是“让插件插入前端组件”，而是让任何模块、任何插件、任何平台内对象，都可以通过统一协议声明：

* 要展示哪些扩展页面 / tab / section
* 每个页面展示什么类型的数据
* 这些数据如何查询
* 支持哪些交互动作
* 前端如何统一渲染

---

# 2. 设计定位

这个系统本质上是一个：

## “平台级 UI 扩展协议 + 后端数据提供协议 + 前端通用渲染引擎”

它位于 XpertAI 架构中的位置应该是：

* 不是 Integration 子模块内部能力
* 不是单纯前端组件库
* 不是低级表单引擎
* 而是 XpertAI Plugin System 和前端业务容器之间的一个标准桥梁层

---

   * 运行输入输出 / step trace / 中间产物 / 依赖资源

# 4. 核心思想

---

## 核心原则 1：扩展的是“视图协议”，不是“组件代码”

插件不直接交付 Angular Component。
插件交付的是：

* View Manifest
* View Schema
* Data Provider
* Action Provider

前端只负责按统一协议渲染。

这保证：

* 插件系统稳定
* 前端体验统一
* 安全边界清晰
* 可审计、可权限控制、可缓存
* 多租户环境下不需要动态前端代码执行

---

## 核心原则 2：平台统一定义有限的 View Type

插件不能无限自定义 UI。
插件只能在平台允许的标准 View Type 中组合：

* stats
* table
* list
* cards
* detail
* kv
* timeline
* chat_messages
* logs
* raw_json
* form
* split
* tabs
* master_detail

这样既有扩展性，又不失控。

---

## 核心原则 3：内容、数据、动作分层

一个扩展视图由三部分组成：

### A. View Manifest

定义有什么 tab / section / page

### B. View Schema

定义怎么显示

### C. View Data

定义显示什么数据

### D. View Action

定义用户在这个视图里可以触发什么动作

---

## 核心原则 4：挂载点由宿主模块决定

不是所有对象都一定要用 tabs。
宿主模块定义它支持的插槽 / 挂载点，例如：

* detail.tabs
* detail.sidepanels
* detail.sections
* dashboard.widgets
* list.row_expansion
* detail.header_actions

插件只是往某个挂载点提供视图。

这样系统就不被 Integration 的 tab 模式绑死。

---

# 5. 总体架构

---

## 5.1 逻辑分层

### 第一层：Host Object

即宿主对象，例如：

* integration
* knowledgebase
* agent
* project
* sandbox
* workflow_run

---

### 第二层：View Slot

宿主暴露可扩展的视图挂载点，例如：

* `detail.main_tabs`
* `detail.secondary_tabs`
* `detail.sections`
* `detail.sidebar`
* `dashboard.widgets`

---

### 第三层：Extension View Manifest

插件或系统模块声明一个或多个扩展视图。

---

### 第四层：Universal View Renderer

前端根据 schema 渲染。

---

### 第五层：Universal View Data API

后端统一查询 schema、data、action。

# 7. 标准视图类型设计

建议第一版不要做无限多，分三层推进。

---

## 7.1 第一阶段必须支持

### 1. stats

适合状态概览、连接情况、计数指标

```ts
export interface XpertStatsViewSchema {
  type: 'stats'
  items: Array<{
    key: string
    label: string
    valueType?: 'text' | 'number' | 'status' | 'datetime'
  }>
}
```

---

### 2. table

最核心的通用视图

```ts
export interface XpertTableViewSchema {
  type: 'table'
  columns: Array<{
    key: string
    label: string
    dataType?: 'text' | 'number' | 'date' | 'datetime' | 'status' | 'tag' | 'avatar' | 'link'
    width?: string
    sortable?: boolean
    searchable?: boolean
  }>
  pagination?: {
    enabled: boolean
    pageSize?: number
  }
  filters?: XpertFilterSchema[]
  rowActions?: XpertRowActionSchema[]
}
```

---

### 3. list

适合简单条目流

```ts
export interface XpertListViewSchema {
  type: 'list'
  item: {
    title: string
    subtitle?: string
    description?: string
    meta?: string[]
  }
  actions?: XpertRowActionSchema[]
}
```

---

### 4. detail

适合单对象详情

```ts
export interface XpertDetailViewSchema {
  type: 'detail'
  fields: Array<{
    key: string
    label: string
    dataType?: 'text' | 'datetime' | 'status' | 'json'
  }>
}
```

---

### 5. raw_json

调试、开发期非常有用

```ts
export interface XpertRawJsonViewSchema {
  type: 'raw_json'
}
```

---

## 7.2 第二阶段推荐支持

### 6. cards

适合用户、频道、资源卡片

### 7. timeline

适合事件流、运行轨迹、同步历史

### 8. chat_messages

适合消息列表、会话消息流

### 9. logs

适合运行日志、Webhook 日志、错误日志

### 10. kv

适合状态、配置摘要展示

---

## 7.3 第三阶段支持复合视图

### 11. tabs

一个扩展视图内部再嵌套 tabs

### 12. split

左右布局

### 13. master_detail

左侧列表 + 右侧详情 / 消息流

这个对 Integration 的 conversation -> messages 特别有用。

# 10. 后端插件协议设计

这里才是整个系统的关键。

不要把它做成 Integration Provider 的附属接口，而是做成平台统一扩展能力。

---

## 10.1 新增平台级插件能力类型

建议新增：

* `view-extension-provider`

它与 Integration 插件、Knowledge 插件、Tool 插件是平级能力。

---

## 10.2 Provider 接口

```ts
export interface IXpertViewExtensionProvider {
  supports(context: XpertViewHostContext): Promise<boolean> | boolean

  getViewManifests(
    context: XpertViewHostContext,
    slot: string
  ): Promise<XpertExtensionViewManifest[]>

  getViewData(
    context: XpertViewHostContext,
    viewKey: string,
    query: XpertViewQuery
  ): Promise<XpertViewDataResult>

  executeViewAction?(
    context: XpertViewHostContext,
    viewKey: string,
    request: XpertViewActionRequest
  ): Promise<XpertViewActionResult>
}
```

---

## 10.3 为什么 supports 很重要

因为不是所有 provider 都适用于所有 host。

例如：

* Slack 插件只支持 `hostType = integration` 且 integration.type = slack
* GitHub 插件可能支持 integration，也支持 project
* Knowledge 扩展 provider 只支持 knowledgebase

---

## 10.4 建议上下文里注入 hostSnapshot

为了避免 provider 每次自己查 host 基础信息，可以在 service 层统一构建：

```ts
export interface XpertResolvedViewHostContext extends XpertViewHostContext {
  hostSnapshot?: Record<string, any>
}
```

例如 integration hostSnapshot 可包含：

* integrationType
* provider
* config
* credentialsRef
* enabled
* status

---

# 11. 后端模块划分建议

建议新增一个独立模块：

## `ViewExtensionModule`

它是平台级公共模块。

---

## 11.1 模块组成

### 1. `view-extension.interfaces.ts`

定义标准接口

### 2. `view-extension.registry.ts`

注册并发现 provider

### 3. `view-extension.service.ts`

负责：

* 解析 host
* 找到适配 provider
* 聚合 manifests
* 查询 data
* 执行动作

### 4. `view-extension.controller.ts`

提供统一 API

### 5. `view-extension-permission.service.ts`

统一权限判断

### 6. `view-extension-cache.service.ts`

schema / data 缓存

---

## 11.2 Registry 设计

你现在已有 strategy registry 体系，很适合新增：

```ts
@XpertPluginStrategy({
  type: 'view-extension-provider',
  name: 'slack-integration-view'
})
```

然后 registry 用：

```ts
Map<string, IXpertViewExtensionProvider>
```

或按组织隔离：

```ts
Map<organizationId, Map<string, IXpertViewExtensionProvider>>
```

# 12. 统一 API 设计

这里要做成平台级，不要写死成 `/integrations/:id/views`。

---

## 12.1 获取某宿主对象某挂载点的扩展视图

```http
GET /api/view-hosts/:hostType/:hostId/slots/:slot/views
```

示例：

```http
GET /api/view-hosts/integration/123/slots/detail.main_tabs/views
GET /api/view-hosts/knowledgebase/456/slots/detail.sections/views
GET /api/view-hosts/agent/789/slots/detail.sidebar/views
```

---

## 12.2 获取某个视图的数据

```http
GET /api/view-hosts/:hostType/:hostId/views/:viewKey/data
```

---

## 12.3 执行某个视图动作

```http
POST /api/view-hosts/:hostType/:hostId/views/:viewKey/actions/:actionKey
```

---

## 12.4 可选：获取 slot 元信息

```http
GET /api/view-hosts/:hostType/:hostId/slots
```

让宿主前端知道有哪些挂载点可用。

宿主模块不需要知道每个插件细节，只需要声明它有哪些扩展插槽。

---

## 13.1 Host Definition 建议

新增 host 定义注册机制：

```ts
export interface XpertViewHostDefinition {
  hostType: string

}
```

例如 Integration 注册：

```ts
{
  hostType: 'integration',
  slots: [
    { key: 'detail.main_tabs', mode: 'tabs' },
    { key: 'detail.sidebar', mode: 'sidebar' }
  ]
}
```

Knowledgebase 注册：

```ts
{
  hostType: 'knowledgebase',
  slots: [
    { key: 'detail.main_tabs', mode: 'tabs' },
    { key: 'detail.sections', mode: 'sections' }
  ]

```

## 13.2 宿主页面前端用法

宿主页面只做两件事：

### 1. 定义自身固定内容

比如 Integration 的 Config Tab

### 2. 拉取某个 slot 的扩展视图并渲染

比如：

* `detail.main_tabs`
* 固定 config tab + 扩展 tabs 合并

# 14. 前端渲染架构

建议把前端做成平台级渲染引擎，而不是某个业务页面内部工具。

---

## 14.1 核心组件层次

```ts
XpertExtensionHostOutletComponent
 ├── receives hostType + hostId + slot
 ├── loads extension view manifests
 ├── renders by slot mode
      ├── tabs renderer
      ├── sections renderer
      ├── sidebar renderer
      └── widgets renderer
           └── XpertViewRendererComponent
                ├── StatsViewComponent
                ├── TableViewComponent
                ├── ListViewComponent
                ├── DetailViewComponent
                ├── RawJsonViewComponent
                ├── TimelineViewComponent
                ├── ChatMessagesViewComponent
                ├── MasterDetailViewComponent
                └── ...
```

---

## 14.2 前端核心对象

```ts
export interface XpertExtensionViewVm {
  key: string
  title: string
  icon?: string
  slot: string
  order?: number
  schema: XpertViewSchema
  actions?: XpertViewActionDefinition[]
  dataSource: XpertViewDataSource

  loaded?: boolean
  loading?: boolean
  data?: XpertViewDataResult
  error?: string
}
```

---

## 14.3 前端服务层

### `XpertViewExtensionApiService`

负责：

* list views
* load data
* execute action

### `XpertViewRendererRegistry`

负责：

* view.type -> Angular renderer component 映射

### `XpertExtensionHostStore`

负责：

* 某个 host 某个 slot 的状态管理

# 15. 渲染引擎设计原则

---

## 原则 1：Renderer 只关心标准 schema

例如 TableRenderer 不知道 Slack、GitHub、Knowledgebase，只知道：

* columns
* items
* actions
* filters
* pagination

---

## 原则 2：容器负责数据调度，Renderer 负责显示

不要让 TableRenderer 自己知道 hostType。
数据请求由上层 container 完成，renderer 只显示数据和抛出事件。

---

## 原则 3：支持 lazy load

tab 模式下只在用户点击时加载对应数据。

---

## 原则 4：支持通用 action pipeline

row action / toolbar action 统一从容器派发到平台 API。

---

# 18. 权限与安全设计

通用能力必须先把安全边界设计清楚。

---

## 18.1 插件不能注入任意前端代码

禁止：

* HTML 字符串执行
* Angular component runtime 注入
* JS 脚本注入
* 自定义模板表达式注入

只允许声明式 schema。

---

## 18.2 所有 action 必须走平台网关

流程必须是：

前端 -> XpertAI API -> ViewExtensionService -> Plugin Provider -> External Service

不能：
前端 -> 插件私有 URL -> 外部服务

---

## 18.3 schema 受平台校验

后端在返回 manifest 之前应做 schema validation。

建议用 `zod` 或 `class-validator + class-transformer` 做标准校验。

---

## 18.4 字段级权限控制

可在字段 schema 中增加：

```ts
permission?: string
sensitive?: boolean
mask?: 'email' | 'phone' | 'token'
```

平台统一处理。

---

# 19. 缓存与性能设计

---

## 19.1 Manifest 缓存

`getViewManifests()` 通常变化较少，可做短时缓存。

建议 key：

```ts
orgId + hostType + hostId + slot + userPermissionHash
```

---

## 19.2 Data 懒加载

只有打开对应 tab / section 才加载 data。

---

## 19.3 支持 polling

例如：

* status 每 30 秒刷新
* logs 每 5 秒刷新
* messages 每 10 秒刷新

由 manifest 的 polling 控制。

---

## 19.4 分页与 cursor

会话、消息、日志必须支持。

---

## 19.5 后端 provider 自己可加二级缓存

比如 Slack users、channels 可以在 provider 内做临时缓存。

---

# 20. 落地实施建议

建议分三期落地，不要一口气做成复杂低代码系统。

---

## Phase 1：最小可用版本

# 22. 示例：Slack Integration Provider

下面给你一版更接近真实落地的简化伪代码。

```ts
@XpertPluginStrategy({
  type: 'view-extension-provider',
  name: 'slack-integration-view'
})
export class SlackIntegrationViewProvider implements IXpertViewExtensionProvider {
  async supports(context: XpertViewHostContext): Promise<boolean> {
    if (context.hostType !== 'integration') return false
    // resolve integration type from hostSnapshot or service
    return true
  }

  async getViewManifests(
    context: XpertViewHostContext,
    slot: string
  ): Promise<XpertExtensionViewManifest[]> {
    if (slot !== 'detail.main_tabs') return []

    return [
      {
        key: 'status',
        title: 'Status',
        hostType: 'integration',
        slot: 'detail.main_tabs',
        order: 10,
        source: { provider: 'slack', plugin: 'xpert-plugin-slack' },
        refreshable: true,
        view: {
          type: 'stats',
          items: [
            { key: 'connectionStatus', label: 'Connection Status', valueType: 'status' },
            { key: 'workspaceName', label: 'Workspace', valueType: 'text' },
            { key: 'lastSyncAt', label: 'Last Sync', valueType: 'datetime' }
          ]
        },
        dataSource: {
          mode: 'remote',
          endpoint: `/api/view-hosts/integration/${context.hostId}/views/status/data`,
          method: 'GET'
        }
      },
      {
        key: 'users',
        title: 'Users',
        hostType: 'integration',
        slot: 'detail.main_tabs',
        order: 20,
        source: { provider: 'slack', plugin: 'xpert-plugin-slack' },
        view: {
          type: 'table',
          columns: [
            { key: 'displayName', label: 'Display Name', dataType: 'text', searchable: true },
            { key: 'email', label: 'Email', dataType: 'text' },
            { key: 'status', label: 'Status', dataType: 'status' }
          ],
          pagination: { enabled: true, pageSize: 20 }
        },
        dataSource: {
          mode: 'remote',
          endpoint: `/api/view-hosts/integration/${context.hostId}/views/users/data`,
          method: 'GET',
          querySchema: {
            supportsPagination: true,
            supportsSearch: true
          }
        }
      }
    ]
  }

  async getViewData(
    context: XpertViewHostContext,
    viewKey: string,
    query: XpertViewQuery
  ): Promise<XpertViewDataResult> {
    switch (viewKey) {
      case 'status':
        return {
          summary: {
            connectionStatus: 'connected',
            workspaceName: 'XpertAI Team',
            lastSyncAt: new Date().toISOString()
          }
        }
      case 'users':
        return {
          items: [],
          total: 0
        }
      default:
        return { items: [] }
    }
  }
}
```

# 23. 最终建议的产品级命名

为了避免它看起来只是个“动态容器”，建议命名更上层一些：

## 后端能力名称

* `View Extension Framework`
* `Declarative View Extension System`

## 前端渲染引擎名称

* `Universal View Renderer`
* `Extension View Renderer`

## 宿主接入组件名称

* `ExtensionHostOutlet`
* `ViewSlotOutlet`

# 24. 一句话架构结论

**不要为 Integration 单独设计 Dynamic Container Renderer，而要把它上升为 XpertAI 平台级的“声明式扩展视图系统”：宿主对象暴露视图插槽，插件或模块声明扩展视图清单、数据源和动作，前端通过统一渲染引擎按标准 schema 渲染，Integration 只是其中一个 host。**
